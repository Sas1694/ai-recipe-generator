# TFM Project Knowledge File

## AI-Powered Recipe Generator Web Application

### Project Overview

This project consists of developing a web application that helps users
generate cooking recipes based on the ingredients they currently have
available. The application integrates modern web technologies and
external AI services.

Users can upload a photo of their fridge, pantry, or ingredients. An AI
vision model detects potential ingredients in the image. The system then
allows the user to review and edit the detected ingredients before
generating recipes using a language model. The system can also generate
an AI image representing the final dish.

The goal is to demonstrate practical integration of AI services within a
modern web architecture while maintaining robustness, usability, and
clear system design.

------------------------------------------------------------------------

# Core Functionalities

## Ingredient Detection from Image

Users upload a photo containing food ingredients. A vision model
analyzes the image and returns a list of detected ingredients.

Because vision models can be inaccurate in complex environments
(fridges, packaging, occlusions), the system implements a
**Human-in-the-Loop approach**.

Flow:

1. User uploads image.
2. Vision model detects possible ingredients.
3. System shows editable ingredient list.
4. User removes incorrect ingredients.
5. User adds missing ingredients.
6. User confirms final ingredient list.

Only the **confirmed ingredients** are used for recipe generation.

Images uploaded by users are **not permanently stored**. They are only
used for temporary analysis.

------------------------------------------------------------------------

## Recipe Generation

After confirming the ingredient list, the user can request recipe
generation.

Flow:

1. Frontend sends confirmed ingredients to backend.
2. Backend constructs prompt for LLM.
3. LLM generates structured recipe output.
4. Recipe is saved to database.
5. Recipe is returned to the user interface.

Recipe includes:

- title
- description
- ingredient list
- step-by-step instructions

------------------------------------------------------------------------

## Dish Image Generation

Users can generate an AI image of the final dish.

Flow:

1. User requests image generation.
2. Backend sends prompt to image generation model.
3. Model returns generated image.
4. Image is stored in storage.
5. Database stores reference to image.
6. Image is displayed to user.

Images generated by AI **are stored**, unlike uploaded images.

# Authentication Strategy

The application uses **Auth.js (NextAuth) with the Credentials Provider**.

Users authenticate using:

- email
- password

Passwords are **never stored in plain text**. Instead, a secure
`passwordHash` is stored in the database using a hashing algorithm such
as bcrypt or argon2.

Session management uses the **JWT session strategy**, meaning sessions
are stored in encrypted cookies rather than in the database.

Because of this architecture:

- No `Session` table is required
- No `Account` table is required
- No `VerificationToken` table is required

Only the **User** entity is persisted for authentication purposes.

This simplifies the database schema while maintaining secure
authentication.

## Image Generation (GPT-image-1-mini)

Image generation cost depends mainly on:

- Image size
- Quality level
- Number of output tokens generated by the model

For images sized 1024×1024 the approximate token usage and cost are:

Quality: low
Output tokens: ~272
Approximate cost: ~$0.002 – $0.003

Quality: medium
Output tokens: ~1056
Approximate cost: ~$0.008 – $0.009

Quality: high
Output tokens: ~4160
Approximate cost: ~$0.033 – $0.035

These estimates are based on OpenAI pricing:

Input tokens: $2.50 per 1M tokens  
Output image tokens: $8.00 per 1M tokens

Because image token generation depends on size and quality, the cost can be estimated before sending the API request.

---

### Simple Cost Estimation Strategy

Applications can estimate image generation cost without inspecting tokens.

Example:

const imageCost = {
  low: 0.0022,
  medium: 0.0085,
  high: 0.033
}

totalCost = imageCost[quality] * numberOfImages

This approach allows backend services to estimate costs before making API calls.

---

### AI Cost Control Strategies

Production AI applications should implement cost control mechanisms to avoid unexpected expenses.

---

#### 1. Logging API Usage

Each OpenAI API response includes usage metadata.

Example:

{
  "usage": {
    "input_tokens": 210,
    "output_tokens": 560
  }
}

Applications should store this information in a database table such as:

Table: ai_usage

Fields example:

user_id  
feature  
model  
input_tokens  
output_tokens  
cost  
created_at  

This enables monitoring:

- Cost per feature
- Cost per user
- Total AI spending

------------------------------------------------------------------------

#### 2. Pre-Request Cost Estimation

Before calling the API, estimate the request cost.

Example:

function estimateImageCost(quality) {
  const prices = {
    low: 0.0022,
    medium: 0.0085,
    high: 0.033
  }

  return prices[quality]
}

This allows applications to enforce usage limits.

------------------------------------------------------------------------

#### 3. Usage Limits Per User

Applications typically define limits depending on subscription plans.

Example:

Free plan
- 5 recipes per day
- low image quality

Pro plan
- 50 recipes per day
- medium image quality

Premium plan
- unlimited usage
- high image quality

------------------------------------------------------------------------

#### 4. AI Response Caching

Many AI requests produce repeated results.

Applications can cache results to avoid repeated API calls.

Example table: recipe_ai_cache

Fields:

hash  
recipe  

Example:

chicken_rice_carrot → generated recipe

If another user requests the same ingredients, the cached recipe can be returned without calling the API.

This technique can reduce costs by 30–70%.

------------------------------------------------------------------------

#### 5. Multi-Stage Quality Generation

A common optimization pattern is to generate a quick preview and only upgrade quality if the user interacts.

Example flow:

Upload ingredient photo  
↓  
Detect ingredients (vision model)  
↓  
Generate recipe (LLM)  
↓  
Generate low-quality image  
↓  
If user saves recipe → regenerate medium image

------------------------------------------------------------------------

#### 6. Internal Cost Dashboard

Production systems often include an internal dashboard to monitor AI spending.

Example metrics:

Daily cost tracking

Monday: $2.30  
Tuesday: $1.90  
Wednesday: $3.10  

Cost by feature

Recipe generation: $40  
Image generation: $28  

This helps detect abnormal usage quickly.

------------------------------------------------------------------------

#### 7. Prompt Size Protection

Applications should limit prompt size to prevent excessive token consumption.

Example:

if(prompt.length > 2000) {
  throw new Error("Prompt too large")
}

## Recipe Generation Model (LLM - gpt-4o-mini)

### Recommended Model Strategy

For recipe generation, the system will use OpenAI language models with a tiered approach:

- Default model: **gpt-4o-mini**
- Premium model: **gpt-4o**

#### Why gpt-4o-mini

- Low cost → scalable
- Fast responses → better UX
- Good enough quality for structured recipe generation
- Strong performance in JSON outputs

#### Why gpt-4o (Premium)

- Higher quality text generation
- Better for complex or gourmet recipes
- Can be enabled for paid users

------------------------------------------------------------------------

### Model Selection Strategy

The model should be abstracted in the service layer:

```ts
function getModelByPlan(plan) {
  if (plan === "premium") return "gpt-4o"
  return "gpt-4o-mini"
}
```

## Ingredient Detection (gpt-4o-mini)

### Overview

The ingredient detection system is responsible for analyzing user-uploaded images and extracting a list of food ingredients.

This component is part of the **ingredient-detection module** and represents the first step in the core user flow.

The system follows an **AI-as-a-Service approach**, using OpenAI multimodal models instead of custom-trained computer vision models.

------------------------------------------------------------------------

### Functional Requirements

#### 1. Detect Food Ingredients in Real Images

The system must handle real-world conditions such as:

- Fridges with multiple items
- Pantry shelves
- Packaged food products
- Occluded or partially visible ingredients
- Complex backgrounds

------------------------------------------------------------------------

#### 2. Return Structured Output

The model must return a normalized list of ingredients:

```json
["egg", "milk", "tomato", "cheese"]
```

Constraints:

- No duplicates
- Lowercase normalized names
- Singular form preferred
- Only edible ingredients

------------------------------------------------------------------------

#### 3. Performance Requirements

The model must be:

- Fast → good user experience  
- Cost-efficient → scalable system  
- Sufficiently accurate → not perfect (Human-in-the-Loop)

------------------------------------------------------------------------

### Model Selection Strategy

#### Default Model: gpt-4o-mini

The system uses **gpt-4o-mini** as the default vision model.

##### Reasons:

- Low cost → enables scalability  
- Fast response times → better UX  
- Native multimodal support (image + text)  
- Good enough accuracy for ingredient detection  

------------------------------------------------------------------------

#### Premium Model: gpt-4o

The system can optionally use **gpt-4o** for higher-tier users.

##### Reasons:

- Better visual understanding  
- Improved accuracy in complex scenes  
- Fewer false positives  

##### Trade-offs:

- Higher cost  
- Higher latency  

------------------------------------------------------------------------

### Model Selection Logic

The model selection should be abstracted in the service layer:

```ts
function getVisionModelByPlan(plan) {
```

# Data Model

## User

Represents an authenticated user of the application.

Fields:

id (uuid)  
name  
email (unique)  
passwordHash  
createdAt

------------------------------------------------------------------------

## Recipe

Represents an AI-generated recipe. Recipes are **shared entities** — they
are not owned by a single user but linked via the UserRecipe join table.

Fields:

id (uuid)  
title  
description  
visualDescription  
ingredientHash (indexed)  
createdAt

**ingredientHash**: deterministic hash of sorted, normalized ingredients.
Before calling the LLM, the system checks if a recipe with the same hash
already exists. If so, the existing recipe is reused — avoiding a new AI
call (estimated 30–70% cost reduction).

**visualDescription**: describes the final plated dish visually, used as
prompt input for AI image generation.

------------------------------------------------------------------------

## UserRecipe

Join table for the **many-to-many** relationship between User and Recipe.

Fields:

id (uuid)  
userId (FK → User)  
recipeId (FK → Recipe)  
savedAt

Unique constraint on (userId, recipeId).

When a user is deleted, only their UserRecipe rows are removed. The Recipe
itself is preserved as cache for other users and future lookups.

------------------------------------------------------------------------

## RecipeIngredient

Represents ingredients required for a generated recipe.

Fields:

id (uuid)  
recipeId (FK → Recipe)  
name  
quantity  
unit

Examples:

- 2 eggs
- 200g pasta
- 1 tbsp olive oil

Ingredients are stored per recipe (no global catalog). Cascade delete
from Recipe.

------------------------------------------------------------------------

## RecipeStep

Represents individual preparation steps of a recipe.

Fields:

id (uuid)  
recipeId (FK → Recipe)  
stepNumber  
instruction

This allows ordered step-by-step cooking instructions. Cascade delete
from Recipe.

------------------------------------------------------------------------

## RecipeImage

Represents an AI-generated image of the dish.

Fields:

id (uuid)  
recipeId (FK → Recipe)  
imageUrl  
prompt  
model  
createdAt

Images are stored in an external storage service, while the database
stores the image reference and generation metadata. Cascade delete
from Recipe.

# Key Architectural Decisions

## Human-in-the-Loop AI

AI suggestions must be confirmed by the user to improve reliability.

## Do Not Store Uploaded Images

User images are only used temporarily for analysis to:

- reduce storage costs
- improve privacy
- simplify architecture

## Persist Results Before Display

Generated recipes and images are stored before being shown to users to
guarantee consistency and avoid displaying data that might later be lost
due to persistence errors.

## Simplified Authentication Model

The system uses Auth.js with the Credentials Provider and JWT sessions.
Only the `User` entity is persisted in the database, reducing schema
complexity while maintaining secure authentication practices.

# Daily Recipe Generation Limit (Cost Control)

To prevent uncontrolled AI costs, the system limits the number of
recipes a user can generate per day.

Each user can generate **a maximum of 5 recipes per day**.

This rule ensures: - Predictable AI usage costs - Protection against
abuse - Fair usage of AI resources

## Validation Flow

Before generating a recipe, the backend checks how many recipes the user
has generated during the current day.

Flow:

1.  User requests recipe generation.
2.  Backend receives request.
3.  Backend queries the database for recipes generated today by the
    user.
4.  If the number of recipes is **\>= 5**, the request is rejected.
5.  If the number is **\< 5**, recipe generation proceeds normally.

Example logic:

    const todayRecipes = await recipeRepository.countRecipesGeneratedToday(userId)

    if (todayRecipes >= 5) {
      throw new Error("Daily recipe generation limit reached")
    }

## Database Query Strategy

The repository can implement a query such as:

    SELECT COUNT(*)
    FROM recipes
    WHERE user_id = $1
    AND created_at >= CURRENT_DATE

This ensures only recipes generated on the same day are counted.

## User Experience

If the limit is reached, the UI should display a message such as:

"Daily recipe limit reached. You can generate more recipes tomorrow."

## Future Extension

The limit can later be adapted depending on the subscription plan:

Free plan: - 5 recipes per day

Pro plan: - 50 recipes per day

Premium plan: - Unlimited recipes

This approach keeps the architecture scalable while maintaining cost
control.

# Technical Decisions

The following technical decisions were made during the planning phase
and must be respected throughout development.

## Directory Structure

The project does NOT use a `src/` directory.

All main directories are placed at the root level:

- `app/` → Presentation layer (Next.js App Router)
- `modules/` → Domain modules (business logic)
- `shared/` → Shared infrastructure and utilities

Path aliases use `@/*` mapping to the project root.

## Database (Local Development)

- Docker Compose with PostgreSQL 16
- Port: 5432
- Database name: `ai_recipe_db`
- Managed via `docker-compose.yml` at project root

## Testing Framework

- Framework: **Vitest**
- Strategy: **TDD** (Test-Driven Development)
- Test scope:
  - use-cases
  - business logic
- Do NOT test:
  - UI components
  - Pages generated with v0
- Test file pattern: `**/*.test.ts`
- Scripts:
  - `npm run test` → watch mode
  - `npm run test:run` → single run (CI)

## Image Storage

- Service: **Vercel Blob**
- Used for storing AI-generated dish images
- Uploaded user images are NOT stored permanently

## Authentication

- Library: **Auth.js v5** (next-auth@beta)
- Provider: Credentials (email + password)
- Session strategy: JWT (no session table in DB)
- Password hashing: bcryptjs

## Environment Variables

Required variables:

- `DATABASE_URL` → PostgreSQL connection string
- `AUTH_SECRET` → Auth.js secret key
- `OPENAI_API_KEY` → OpenAI API key
- `BLOB_READ_WRITE_TOKEN` → Vercel Blob access token

## UI Component Library

- **shadcn/ui** with Tailwind CSS v4
- Mobile-first design approach
- UI generation assisted by v0

## Input Validation

- Library: **Zod**
- Applied in Server Actions (actions layer)
